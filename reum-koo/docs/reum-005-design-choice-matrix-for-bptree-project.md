# reum-005 B+ 트리 프로젝트 설계 선택지와 추천 조합

## 이 문서의 역할

이 문서는 설계 대화 export 안에 흩어져 있던 **선택지, 장단점, 추천안**을 한 문서로 재정리한 것이다.

핵심 질문은 이거다.

> 요구사항은 이미 정해져 있다.  
> 그렇다면 우리는 어떤 구조를 골라서 구현할 것인가?

---

## 1. 먼저 구분해야 할 세 가지: 요구사항, 설계, 구현

설계 대화에서 가장 먼저 풀어준 핵심 중 하나가 바로 이 구분이다.

### 표 1. 세 개념의 차이

| 구분 | 뜻 | 예시 |
| --- | --- | --- |
| 요구사항 | 과제가 이미 정해놓은 것 | 자동 ID, `WHERE id = ?` 인덱스 사용 |
| 설계 | 그 요구사항을 어떤 구조로 만들지 정하는 것 | `row_index`를 value로 쓸지, 포인터를 쓸지 |
| 구현 | 설계한 내용을 C 코드로 쓰는 것 | `typedef struct Table ...`, `bptree_insert()` |

### 왜 이 구분이 중요한가

많이 헷갈리는 지점은 이거다.

- "이건 꼭 해야 하는 건가?"
- "이건 우리 팀이 정하는 건가?"

답은:

- 요구사항은 바꿀 수 없다
- 설계는 팀이 고른다
- 구현은 그 설계를 코드로 쓰는 것이다

---

## 2. 팀장이 말한 “정하자”의 진짜 뜻

대화 export에서는 팀장의 말을 이렇게 번역했다.

> 과제 요구사항은 이미 정해져 있다.  
> 그런데 그 요구사항을 만족시키는 방법은 여러 개이므로,  
> 우리 팀은 어떤 방식으로 만들지 먼저 고정하자.

즉, 팀장이 말한 "정하자"는 말은 보통 이런 항목들을 뜻한다.

### 표 2. 회의에서 실제로 정해야 하는 것

| 항목 | 질문 |
| --- | --- |
| 테이블 저장 방식 | 레코드를 어디에 둘까 |
| 인덱스 value 의미 | 트리가 실제 데이터를 어떻게 가리킬까 |
| 검색 경로 | `WHERE id = ?`면 어떤 순서로 데이터를 찾을까 |
| SQL 연동 위치 | parser에서 할까, executor에서 할까 |
| 노드 구조 | 하나의 struct로 할까, leaf/internal을 나눌까 |
| split 정책 | separator key를 어떤 규칙으로 올릴까 |

이런 빈칸을 채우는 일이 설계다.

### 설계 회의에서 초보자가 자주 헷갈리는 지점

설계 회의에서는 종종 서로 다른 종류의 말을 섞어버린다.

| 섞이기 쉬운 말 | 사실은 어떤 차이인가 |
| --- | --- |
| "이건 과제에서 꼭 해야 하나?" | 요구사항 질문 |
| "이건 배열로 할까, 포인터로 할까?" | 설계 질문 |
| "그럼 함수 이름은 뭘로 할까?" | 구현 질문 |

이 셋을 분리하면 회의가 훨씬 빨라진다.
예를 들어 "`WHERE id = ?`에서 인덱스를 써야 하나?"는 요구사항에 가깝고,
"인덱스 value를 `row_index`로 둘까 `Record*`로 둘까?"는 설계에 가깝다.

## 3. 가장 먼저 고를 것: 테이블과 인덱스를 어떻게 붙일까

설계 대화에서 가장 큰 갈림길로 제시된 부분이다.

### 방식 A. clustered 비슷하게 가는 방식

리프 노드에 `id`와 함께 **레코드 전체**를 저장하는 방식이다.

#### 장점

- `WHERE id = ?`가 아주 빠르게 끝남
- 별도 테이블 저장소 역할이 약해도 됨

#### 단점

- 리프 split 때 레코드 자체가 움직여서 무거워짐
- 다른 필드 선형 탐색 요구와 잘 안 맞음
- 기존 SQL 처리기와 연결 시 "테이블" 개념이 흐려짐

---

### 방식 B. 저장소와 인덱스를 분리하는 방식

`records[]` 같은 저장소를 따로 두고,
B+ 트리에는 `id -> row_index` 또는 `id -> locator`만 저장하는 방식이다.

#### 장점

- 구현이 단순함
- `WHERE id = ?`와 다른 조건 조회를 자연스럽게 분리 가능
- 기존 SQL 처리기와 연결하기 쉬움
- README 설명이 쉬움

#### 단점

- 인덱스를 찾은 뒤 저장소를 한 번 더 봐야 함

### 표 3. 두 방식 비교

| 항목 | clustered 비슷한 방식 | 분리형 방식 |
| --- | --- | --- |
| 인덱스 리프 | 레코드 전체에 가까움 | 위치 정보만 저장 |
| 구현 난이도 | 높음 | 낮음 |
| 이번 과제 적합성 | 보통 과함 | 매우 적합 |
| 선형 탐색 비교 | 설명이 덜 직관적 | 설명이 매우 쉬움 |

### 이번 과제 추천

설계 대화의 추천은 명확했다.

> 이번 과제는 **저장소와 인덱스를 분리하는 방식**이 가장 안전하다.

## 4. 레코드를 메모리에 어떻게 저장할까

설계 대화에서 나온 선택지는 다음과 같았다.

### 표 4. 저장 구조 선택지

| 선택지 | 장점 | 단점 |
| --- | --- | --- |
| 동적 배열 | 단순함, 선형 탐색 쉬움, 100만 건 적재 쉬움 | `realloc()` 시 주소 변동 가능 |
| 연결 리스트 | 포인터 안정성 | 임의 접근이 느림 |
| slotted page 비슷한 구조 | DB스럽다 | 학습/구현 난이도 상승 |
| 오브젝트 풀/해시 구조 | 특수 목적에 유리 | 과제 범위에 비해 과함 |

### 추천안

> `Record *records` 동적 배열이 가장 무난하다.

이유:

- 대량 삽입이 쉽고
- 선형 탐색이 간단하고
- `row_index`와 연결하기 좋고
- 디버깅이 쉽다

---

## 5. B+ 트리 리프 value는 무엇으로 둘까

이건 설계 대화 전체에서 가장 자주 등장한 질문 중 하나다.

### 표 5. 대표 선택지

| 선택지 | 설명 |
| --- | --- |
| `Record*` | 메모리 주소를 직접 저장 |
| `row_index` | 배열에서 몇 번째 row인지 저장 |
| 레코드 전체 | 실제 데이터를 리프에 통째로 저장 |
| locator | `file_path + offset` 또는 `offset` 같은 위치 정보 저장 |

이 문서에서는 큰 그림만 다루고,
자세한 차이는 `reum-006`에서 다룬다.

### 이번 과제 추천

> 메모리 기반 구현이라면 `id -> row_index`가 가장 단순하고 안전하다.

### 초보자용 판단 질문 4개

설계 선택지가 여러 개 보일 때는 아래 네 질문으로 걸러보면 좋다.

1. 이 구조를 팀원이 말로 설명하기 쉬운가
2. 디버깅할 때 상태를 눈으로 추적하기 쉬운가
3. 기존 SQL 처리기와 붙이기 쉬운가
4. 테스트를 자동화하기 쉬운가

`row_index` 방식은 이 네 질문에 모두 꽤 강하다.
반면 `Record*` 방식은 당장 편해 보여도,
메모리 재할당이나 주소 안정성 문제 때문에 설명과 디버깅이 어려워질 수 있다.

## 6. B+ 트리 노드 모양은 어떻게 정할까

설계 대화에서 나온 두 대표 선택지는 아래와 같다.

### 선택지 1. leaf/internal을 다른 struct로 분리

#### 장점

- 역할이 명확하다

#### 단점

- 코드가 길어지고
- 분기/캐스팅이 늘어난다

---

### 선택지 2. 하나의 struct에 `is_leaf` 플래그 두기

예:

```c
typedef struct BPTreeNode {
    int is_leaf;
    int num_keys;
    int64_t keys[ORDER - 1];
    struct BPTreeNode *children[ORDER];
    int values[ORDER - 1];
    struct BPTreeNode *next;
} BPTreeNode;
```

#### 장점

- 구현이 짧다
- 함수 시그니처가 단순하다

#### 단점

- 내부/리프 분기 코드가 들어간다

### 이번 과제 추천

> 하나의 struct + `is_leaf` 플래그가 실습 과제에는 더 적합하다.

---

## 7. 차수(order)와 leaf 연결은 어떻게 할까

### 표 6. 관련 결정

| 항목 | 추천 |
| --- | --- |
| 차수(order) | 우선 컴파일 타임 상수로 |
| 디버깅 시 order | 4~5처럼 작게 |
| 성능 측정 시 order | 32 또는 64 정도 |
| 리프 연결 | `next` 포인터 유지 |

### 왜 이렇게 추천하나

- 작은 order는 split이 눈에 잘 보여 디버깅이 쉽다
- 큰 order는 성능 실험에 유리하다
- leaf `next`는 나중에 range scan, 시각화, `BETWEEN` 확장에 도움 된다

---

## 8. 삽입 알고리즘은 어떤 스타일로 짤까

설계 대화에서 나온 갈림길은 두 가지다.

### 방식 A. 모든 노드에 parent 포인터 둠

#### 장점

- split 후 부모 갱신이 직관적

#### 단점

- 노드 생성/분할 때 parent 유지가 귀찮다

---

### 방식 B. 재귀 반환으로 split 결과를 부모로 올림

예:

- split 안 나면 “변화 없음” 반환
- split 나면 “승격 key + 새 오른쪽 노드” 반환

#### 장점

- parent 포인터 없이 가능
- C에서도 생각보다 깔끔함

#### 단점

- 반환 구조체 설계가 필요함

### 표 7. 두 방식 비교

| 항목 | parent 포인터 | 재귀 반환 |
| --- | --- | --- |
| 직관성 | 처음엔 쉬움 | 익숙해지면 깔끔함 |
| 상태 관리 | 노드마다 추가 관리 필요 | 반환 구조 설계 필요 |
| 이번 과제 추천 | 보통 덜 추천 | 추천 |

### split 정책에서 반드시 정해야 할 것

| 항목 | 예시 |
| --- | --- |
| 리프 split 후 부모에 올릴 key | 오른쪽 새 리프의 첫 key |
| 내부 split 시 승격 key | 가운데 key |
| root split 시 처리 | 새 root 생성 |

이 규칙을 먼저 정하지 않으면 삽입 코드가 꼬이기 쉽다.

## 9. ID 정책은 어떻게 잡을까

설계 대화에서는 다음 방향을 추천했다.

### 표 8. ID 정책 추천안

| 항목 | 추천 |
| --- | --- |
| 타입 | `int64_t` |
| 증가 방식 | 단조 증가 auto-increment |
| 중복 | 불허 |
| 외부 입력 ID | 이번 과제에서는 비허용이 더 단순 |

### 왜 `int64_t`인가

- 설명이 쉽고
- 대용량 테스트에서 안전하고
- 실무식 자동 증가 PK 감각과 비슷하기 때문이다

## 10. SQL 처리기와 어디서 연결할까

이 부분은 시간이 많이 새기 쉬운 지점이다.

### 방식 A. parser/AST를 크게 확장

#### 장점

- 구조적으로 예쁠 수 있다

#### 단점

- 시간이 많이 든다
- 과제 핵심보다 파서 수정이 커질 수 있다

---

### 방식 B. executor에서 `WHERE id = 정수`만 감지

#### 장점

- 가장 빠르다
- 기존 코드 재사용이 쉽다
- 발표 때 설명이 쉽다

#### 단점

- 범용 SQL 처리기로 가기엔 제한적이다

### 이번 과제 추천

> executor 분기 방식이 가장 적합하다.

### 이유

- 요구사항이 `WHERE id = ?`에 집중되어 있고
- 다른 조건은 선형 탐색으로 남겨도 되며
- SQL 처리기 전체를 다시 짜지 않아도 되기 때문이다

### 이 설계 선택이 실제 코드에서 보이는 위치

설계는 추상적인 말로 끝나지 않는다.
결국 아래 같은 코드 모양으로 드러난다.

| 설계 판단 | 코드에서 보이는 모습 |
| --- | --- |
| 저장소와 인덱스를 분리 | `Table` 안에 `records[]`와 `bptree_index`가 따로 존재 |
| value를 `row_index`로 사용 | 리프 value 타입이 정수 인덱스가 됨 |
| executor에서 분기 | `WHERE id = ?` 여부를 실행기에서 확인 |
| 자동 ID 정책 사용 | `next_id` 같은 상태가 테이블 또는 인덱스 계층에 존재 |

즉, 설계 문서를 읽을 때는
"이 판단이 코드에서는 어떤 필드, 어떤 함수, 어떤 반환값으로 나타날까?"
를 같이 떠올리면 훨씬 빨리 익숙해진다.

## 11. 대량 적재는 SQL 경유가 좋을까, 내부 API가 좋을까

설계 대화에서 이 질문도 중요한 설계 포인트로 다뤄졌다.

### 표 9. 두 방식 비교

| 방식 | 장점 | 단점 |
| --- | --- | --- |
| SQL 100만 건 | SQL 연동을 강하게 보여줌 | 파싱 비용이 커짐 |
| 내부 API 대량 생성 | 저장소/인덱스 성능을 더 정확히 보기 좋음 | 모든 적재가 SQL 경유는 아님 |

### 추천안

- 기능 데모는 SQL 경유
- 성능 벤치마크는 내부 API 경유

즉, 기능 검증과 성능 측정을 분리하는 것이 더 정직하다.

## 12. 성능 비교는 무엇과 무엇을 비교할까

설계 대화에서는 비교 대상을 성격 맞게 잡으라고 했다.

### 좋은 비교

- 랜덤 `id` equality 조회 10,000회
- 랜덤 `name` 또는 다른 비-id equality 조회 10,000회

### 안 좋은 비교

- `WHERE id = ?` 한 번
- `WHERE name = ?` 한 번

### 표 10. 비교 설계 기준

| 기준 | 설명 |
| --- | --- |
| equality vs equality | 비교 조건의 성격을 맞춘다 |
| 반복 측정 | 표본이 너무 적지 않게 한다 |
| 인덱스 유무만 달라지게 | 논점이 흐려지지 않게 한다 |

## 14. 설계안 A/B/C로 보면 더 쉬운 이유

설계 대화에서는 같은 요구사항에도 여러 설계안이 가능하다고 설명했다.

### 설계안 A

- 테이블: 동적 배열
- value: `row_index`
- ID 검색: 인덱스에서 `row_index` 찾고 배열 접근
- 다른 필드 검색: 전체 선형 탐색

이게 추천안이다.

### 설계안 B

- 테이블: 레코드 하나씩 `malloc`
- value: `Record*`
- ID 검색: 인덱스에서 포인터 바로 반환

가능하지만 관리 부담이 있다.

### 설계안 C

- 레코드 전체를 B+ 트리 리프에 저장
- 사실상 인덱스와 저장소를 합침

가능하지만 이번 과제에는 무겁다.

### 표 11. 세 설계안 비교

| 설계안 | 구현 난이도 | 설명 난이도 | 이번 과제 적합성 |
| --- | --- | --- | --- |
| A | 낮음 | 낮음 | 매우 높음 |
| B | 중간 | 중간 | 보통 |
| C | 높음 | 높음 | 낮음 |

## 15. 설계를 먼저 안 정하면 생기는 문제

설계 대화에서는 이 문제도 분명히 짚었다.

예:

- A는 `value = Record*`라고 생각하고 코딩
- B는 `value = row_index`라고 생각하고 코딩
- C는 배열 저장소를 만듦
- D는 연결 리스트를 만듦

그러면 나중에 코드가 서로 안 맞는다.

즉, 설계는 멋있는 문서 작성을 위한 것이 아니라

> 팀원들이 같은 구조를 상상하도록 만드는 약속

이다.

## 16. 이번 과제에서 추천하는 최종 선택 조합

설계 대화 export의 추천 조합을 그대로 정리하면 다음과 같다.

이 섹션은 사실 "정답 발표"라기보다
"이번 과제 조건에서 가장 실용적인 조합"을 모아둔 것이다.
왜냐하면 설계는 절대적인 정답 하나만 있는 경우보다
트레이드오프가 있는 경우가 훨씬 많기 때문이다.

초보자에게 특히 중요한 포인트는 이거다.

- 멋져 보이는 구조보다 설명 가능한 구조가 더 좋다
- 범용적인 구조보다 디버깅 쉬운 구조가 더 좋다
- 확장성이 조금 덜해도 오늘 완성 가능한 구조가 더 좋다

```text
[저장 방식]
records 동적 배열 + B+ 트리(id -> row_index)

[ID 정책]
int64_t auto-increment, unique only

[노드 구조]
단일 struct + is_leaf + next leaf

[차수]
고정 상수(order 32 정도)
디버깅 때는 4~5로 낮춰도 됨

[삽입 방식]
재귀 + split 결과 반환

[SQL 연동]
executor에서 WHERE id = 정수 패턴만 인덱스로 분기

[성능 테스트]
대량 데이터는 내부 API로 생성
기능 데모는 SQL로 수행

[검증]
단위 테스트 + 기능 테스트 + invariant 검사
```

### 이 조합의 장점

| 장점 | 설명 |
| --- | --- |
| 구현 속도 | 빠르다 |
| 요구사항 적합성 | 매우 높다 |
| README 설명 | 쉽다 |
| 확장성 | `BETWEEN`, `EXPLAIN`, `DELETE` 등으로 확장 가능 |

## 17. 팀 회의에서 어떻게 말하면 좋나

설계 대화에서는 실제 회의용 문장도 제안했다.

예:

> 레코드는 동적 배열에 저장하고,  
> B+ 트리 리프 value에는 `row_index`를 넣자.  
> 그러면 `WHERE id = ?`는 인덱스에서 `row_index`를 찾고,  
> 실제 레코드는 `records[row_index]`로 가져오면 된다.

이 정도 문장으로도 팀 설계 방향을 꽤 분명하게 공유할 수 있다.

## 18. 핵심만 다시 압축하면

### 표 12. 이 문서의 핵심

| 질문 | 답 |
| --- | --- |
| 설계에서 제일 먼저 고를 것 | 저장소와 인덱스를 분리할지 여부 |
| 이번 과제 추천 저장소 | 동적 배열 |
| 추천 인덱스 value | `row_index` |
| 추천 노드 표현 | 단일 struct + `is_leaf` |
| 추천 split 스타일 | 재귀 반환 방식 |
| 추천 SQL 연동 | executor 분기 |
| 추천 테스트 | 기능 + invariant |

가장 중요한 한 줄은 이것이다.

> 이번 과제에서는  
> **records 동적 배열 + B+ 트리 `id -> row_index` + executor 분기**
> 조합이 가장 단순하고 안전하다.
