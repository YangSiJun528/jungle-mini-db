# reum-000 문서 학습 로드맵

이 문서는 `reum-koo/docs` 안 문서들을

- 어떤 순서로 읽으면 좋은지
- 어떤 문서는 먼저 읽지 않는 게 좋은지
- 지금 내 목적에 맞는 경로가 무엇인지

를 한눈에 보이게 정리한 **문서 길잡이**다.

문서 번호가 있다고 해서
무조건 `001 -> 002 -> 003 ...` 순서로 읽는 것이 가장 좋은 것은 아니다.

특히 지금 문서 묶음은

- 요구사항 정리 문서
- 입문 배경지식 문서
- 설계 문서
- 코드 흐름 문서
- HTML 비교 리포트

가 섞여 있어서,
읽는 목적에 따라 순서를 바꾸는 편이 훨씬 낫다.

---

## 1. 가장 추천하는 기본 순서

정말 처음부터 이해하고 싶다면 아래 순서가 가장 좋다.

1. `reum-000`
2. `reum-002`
3. `reum-012`
4. `reum-013`
5. `reum-014`
6. `reum-006`
7. `reum-007`
8. `reum-005`
9. `reum-004`
10. `reum-001`
11. `reum-008`
12. `reum-010`

이 순서가 좋은 이유는 이렇다.

- `reum-002`는 DB/SQL 처리기 기본 단어를 잡아준다
- `reum-012`는 B-tree 파라미터를 이해하게 해준다
- `reum-013`은 발표에서 나올 더 넓은 용어를 대비하게 해준다
- `reum-014`는 기술끼리, 제품끼리 무엇을 비교해야 하는지 정리해준다
- `reum-006`, `reum-007`은 프로젝트 코드에 바로 연결되는 개념을 잡아준다
- `reum-005`, `reum-004`는 설계와 구현 순서를 정리해준다
- `reum-001`, `reum-008`, `reum-010`은 비교와 변화, 실험 결과를 읽게 해준다

즉,

> 기초 단어 -> 트리 파라미터 -> 심화 용어 -> 저장/흐름 -> 설계 -> 구현 계획 -> 비교/결과

순서라고 생각하면 된다.

2. `reum-002`
3. `reum-012`
4. `reum-013`
5. `reum-014`
6. `reum-006`

여기까지 읽으면
다른 문서에서 나오는 단어 때문에 막히는 일이 훨씬 줄어든다.

### 경우 B. 우리 프로젝트 코드가 어떻게 도는지 알고 싶은 경우

2. `reum-002`
3. `reum-006`
4. `reum-007`
5. `reum-005`

### 경우 C. 발표에서 다른 팀 설명도 알아듣고 싶은 경우

2. `reum-002`

4. `reum-013`
5. `reum-014`
6. `reum-010`

이 경로는
용어와 비교 관점을 빨리 잡는 데 좋다.

### 경우 D. 구현 계획과 설계만 빨리 정리하고 싶은 경우

1. `reum-000`

이 경로는 회의용으로 좋다.

## 4. 먼저 읽지 않는 게 좋은 문서

아래 문서는 처음부터 바로 들어가면 힘들 수 있다.

### `reum-001`

이 문서는
"이미 어느 정도 구조를 알고 있는 상태"를 전제로 한다.

즉 차이를 보여주는 문서이지,
처음 개념을 잡아주는 문서는 아니다.

### `reum-008`

이 문서는 코드 비교 HTML이라서
기초 개념 없이 보면 코드 줄만 많아 보일 수 있다.

### `reum-010`

이 문서는 벤치마크 결과 HTML이기 때문에

- ops/s
- scenario
- sparse/dense
- lookup winner

같은 말을 모르면 이해가 어렵다.

그래서 `reum-002`, `reum-012`, `reum-013` 이후에 보는 것이 좋다.

## 6. 지금 가장 중요한 추천

지금처럼

> "`reum-002`를 읽었는데 다른 문서가 아직 어렵다"

고 느껴진다면,
다음 순서로 바로 이어서 읽는 것이 가장 좋다.

1. `reum-012`
2. `reum-013`
3. `reum-014`
4. `reum-006`
5. `reum-007`

즉,

> 기본 용어 -> 트리 파라미터 -> 심화 발표 용어 -> 기술/제품 비교 -> 우리 코드 개념

순서로 한 번 더 다져야 한다.
