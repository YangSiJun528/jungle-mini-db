# reum-003 설계 대화 문서 맵과 읽기 순서

## 이 문서는 왜 필요한가

설계 단계에서 GPT와 나눈 대화 export에는 내용이 많다.

- 오늘 하루 구현 순서
- MVP 범위
- 설계 선택지
- 저장 방식과 인덱스 value 후보
- `row_index`, `offset`, `locator` 설명
- 메모리 기반 구현 흐름
- 성능 비교와 테스트 전략
- README와 발표 포인트

이걸 한 파일에 다 넣으면 읽기 어렵다.
그래서 이 문서는 **어떤 내용이 어느 문서로 재정리되었는지 안내하는 지도** 역할을 한다.

## 문서별 역할을 더 자세히 보면

### `reum-004`

이 문서는 "오늘 하루 안에 어떻게 구현을 끝낼 것인가"에 초점을 둔다.

- 범위 고정
- 구현 순서
- 검증 순서
- 성능 측정 방식
- README 정리 방식

즉, **프로젝트 운영 계획 문서**에 가깝다.

### `reum-005`

이 문서는 "어떤 설계를 선택할 것인가"에 초점을 둔다.

- 저장소와 인덱스를 분리할지
- 노드 구조를 어떻게 할지
- split 코드를 어떤 스타일로 짤지
- SQL 처리기와 어디서 연결할지

즉, **설계 의사결정 문서**에 가깝다.

### `reum-006`

이 문서는 가장 헷갈리기 쉬운 개념들만 따로 모은다.

- `Record*`
- `row_index`
- `offset`
- `path + offset`
- locator
- fixed row

즉, **주소/위치/저장 형식 개념 문서**다.

### `reum-007`

이 문서는 코드가 실제로 어떻게 움직이는지 설명한다.

- INSERT 때 무슨 일이 일어나는지
- `WHERE id = ?`에서 인덱스를 어떻게 타는지
- 왜 `WHERE name = ?`는 느린지

즉, **실행 흐름 문서**다.

## 이 문서 묶음을 읽을 때 주의할 점

대화 export에는 세 종류의 내용이 섞여 있다.

### 1. 요구사항

과제가 이미 정해놓은 것.

예:

- 자동 ID
- `WHERE id = ?` 인덱스 사용
- 비-id 선형 탐색
- 100만 건 이상 삽입

### 2. 설계 선택지

요구사항을 만족하기 위한 여러 후보.

예:

- `Record*` vs `row_index`
- 배열 vs 리스트
- executor 분기 vs parser 확장

### 3. 추천안

대화 속 GPT가 이번 과제에 맞다고 본 방향.

예:

- records 동적 배열
- `id -> row_index`
- executor에서 `WHERE id = 정수`만 분기

이 셋을 구분해서 읽어야 헷갈리지 않는다.
