# reum-012 B-tree 파라미터 입문서

이 문서는 아래 같은 말이 나올 때

- 블록 크기(`block size`)
- 페이지 크기(`page size`)
- 최대 키값(`max key`)
- 키 타입(`int`, `int64_t`)
- 차수(`order`)
- fanout
- 한 노드에 들어가는 최대 key 개수
- offset 크기

"대체 왜 이런 걸 정해야 하지?"라는 질문에 답하기 위해 만든
**완전 입문자용 설명서**다.

---

## 1. 먼저 결론부터

B-tree나 B+ tree를 만들 때는
아무 숫자나 막 정하는 게 아니다.

아래가 서로 연결되어 있기 때문이다.

```text
블록 크기
-> 한 노드에 들어갈 수 있는 데이터 양
-> 한 노드에 들어갈 수 있는 key 개수
-> 트리의 높이
-> split 빈도
-> 탐색/삽입 성능
-> 디스크 I/O와 메모리 사용량
```

즉, 파라미터 하나를 바꾸면
트리의 구조와 성능이 같이 바뀐다.

---

## 2. 비유로 먼저 이해해보자

### 블록 크기 = 서랍 한 칸 크기

트리의 각 노드를
"서랍 한 칸"이라고 생각해보자.

- 블록 크기가 크다
  - 서랍 한 칸이 크다
  - key를 많이 넣을 수 있다
  - 트리 높이가 낮아질 수 있다
- 블록 크기가 작다
  - 서랍 한 칸이 작다
  - key를 적게 넣는다
  - 트리 높이가 더 높아질 수 있다

### key 크기 = 서랍 안 물건 하나의 크기

같은 서랍이라도
물건이 크면 적게 들어가고,
물건이 작으면 많이 들어간다.

즉,

- key가 크면 노드당 key 수가 줄어든다
- value가 크면 노드당 key 수가 줄어든다
- child pointer나 offset이 크면 노드당 key 수가 줄어든다

---

## 3. 가장 자주 나오는 용어

| 용어 | 쉬운 뜻 |
| --- | --- |
| `block size` | 노드 하나를 저장할 수 있는 고정 크기 |
| `page size` | 디스크나 OS가 다루는 기본 저장 단위. block size와 비슷하게 맞추는 경우가 많다 |
| `key` | 찾기 위한 기준값. 예: `id` |
| `value` | key로 찾았을 때 얻는 값. 예: row 위치 offset |
| `offset` | 파일 안에서 몇 번째 바이트인지 나타내는 위치 정보 |
| `order` | 내부 노드가 가질 수 있는 자식 수 관련 기준값 |
| `fanout` | 한 노드가 뻗어 나갈 수 있는 가지 수 |
| `max entries` | 리프 노드에 들어갈 수 있는 key-value 쌍 최대 개수 |
| `split` | 노드가 꽉 차서 둘로 나뉘는 일 |
| `height` | 루트에서 리프까지 몇 층인지 |

---

## 4. 왜 블록 크기를 정해야 하나

디스크 기반 구조에서는
노드 하나를 파일에 저장해야 한다.

그런데 노드마다 크기가 제각각이면
읽고 쓰기가 복잡해진다.

그래서 보통은

> "노드 하나는 몇 바이트짜리 블록 하나에 들어간다"

라고 정한다.

예를 들어:

- `4096 bytes`
- `8192 bytes`
- `16384 bytes`

같은 값이 자주 보인다.

이 숫자들은 흔히
OS 페이지 크기나 파일 시스템/DB 페이지 크기 감각과 맞물린다.

### 블록 크기가 커지면

- 한 노드에 key를 더 많이 넣을 수 있다
- 트리 높이가 낮아질 수 있다
- 탐색 시 방문 노드 수가 줄 수 있다
- 하지만 노드 하나 읽을 때 가져오는 데이터 양은 커진다

### 블록 크기가 작아지면

- 노드당 key 수가 줄어든다
- split이 더 자주 일어날 수 있다
- 트리 높이가 올라갈 수 있다
- 대신 작은 단위로 다루기 쉽다

---

## 5. 왜 최대 키값과 key 타입을 정해야 하나

트리는 key를 저장하고 비교한다.

그러면 key 타입부터 정해야 한다.

예:

- `int`
- `long`
- `int64_t`
- 문자열

### `int`를 쓰면

대개 32-bit signed 정수 범위를 가진다.

즉 대략:

```text
-2,147,483,648 ~ 2,147,483,647
```

이 범위를 넘는 key는 안전하지 않다.

### `int64_t`를 쓰면

더 훨씬 큰 수까지 담을 수 있다.

대신 보통 key 크기가 커져서
노드당 key 수가 줄어들 수 있다.

즉 여기서도 trade-off가 생긴다.

```text
큰 key 범위
<-> 더 큰 key 크기
<-> 한 노드에 들어가는 개수 감소 가능
```

---

## 6. value 크기도 왜 중요하나

초보자가 많이 놓치는 부분이 이거다.

트리는 key만 저장하는 게 아니라,
리프 노드에서는 보통 value도 같이 저장한다.

예를 들면:

- `id -> row offset`
- `id -> row_index`
- `id -> pointer`

그러면 노드 안에 들어가는 건 실제로는

```text
key + value
```

쌍이다.

그래서 value가 커지면
리프 노드에 들어갈 수 있는 entry 개수도 줄어든다.

### 예시

블록 크기 = `4096`

만약:

- key = 4바이트
- value = 8바이트

라면
entry 하나가 대략 `12바이트` 수준이다.

그럼 대략 수백 개 정도를 넣을 수 있다.

그런데 key와 value가 둘 다 더 커지면
같은 4096바이트 안에 들어가는 개수는 줄어든다.

---

## 7. 차수(order), fanout, max entries는 뭔 차이인가

이 부분이 특히 헷갈린다.

### fanout

한 내부 노드가
최대 몇 개 자식을 가질 수 있는가를 뜻한다.

즉 "가지 수"다.

### order

문헌이나 구현마다 뜻이 조금 다르다.

그래서 이 단어는 항상
"이 코드에서 order를 어떻게 정의했는지"
를 같이 봐야 한다.

어떤 구현에서는

- 최대 자식 수를 order라고 부르고

어떤 구현에서는

- 최소 차수(minimum degree)를 order라고 부른다.

즉 `order`는
**정의가 통일된 단어가 아니다**.

### max entries

리프 노드 하나에 들어갈 수 있는
key-value 쌍 최대 개수다.

이건 비교적 직관적이다.

---

## 8. 한 노드에 몇 개 들어가나: 왜 공식이 나오나

노드 안에는 순수하게 key만 있는 게 아니다.

보통 이런 것들이 같이 들어간다.

- 부모 위치
- 형제 위치
- 현재 children 개수
- node type
- key 배열
- value 배열 또는 child pointer 배열

즉,

```text
노드 전체 크기
= 헤더 + key 배열 + value/child 배열
```

그래서 공식이 나온다.

### 아주 단순화한 느낌의 공식

리프 노드라면:

```text
max_entries
= (block_size - header_size) / (sizeof(key) + sizeof(value))
```

내부 노드라면:

```text
max_order
= (block_size - header_size) / (sizeof(key) + sizeof(child_pointer))
```

정확한 공식은 구현마다 조금씩 다르지만,
큰 그림은 거의 이렇다.

---

## 9. 이 프로젝트의 `thirdparty/bplustree`에서는 어떻게 계산하나

실제 코드에는 이런 계산이 있다.

```c
_max_order = (block_size - sizeof(node)) / (sizeof(key_t) + sizeof(off_t));
_max_entries = (block_size - sizeof(node)) / (sizeof(key_t) + sizeof(long));
```

이 말은:

- 내부 노드 최대 크기 계산에는
  `key_t`와 `off_t`를 쓴다
- 리프 노드 최대 크기 계산에는
  `key_t`와 `long`을 쓴다

는 뜻이다.

즉 여기서는

- 내부 노드: `key + child offset`
- 리프 노드: `key + data`

구조를 가진다고 이해하면 된다.

---

## 10. `off_t`는 또 뭐야

`off_t`는 파일 안 위치를 나타낼 때 많이 쓰는 타입이다.

즉:

```text
"파일에서 몇 바이트 떨어져 있나"
```

를 표현할 때 자주 쓴다.

그래서 디스크 기반 인덱스 구현에서는
child pointer 대신

```text
child offset
```

을 들고 있는 경우가 많다.

메모리 기반 트리는 보통

- 진짜 포인터

를 들고 있을 수 있지만,

디스크 기반 트리는 보통

- 파일 offset

을 들고 있게 된다.

---

## 11. 블록 크기를 바꾸면 실제로 뭐가 달라지나

### 4096 -> 8192로 키우면

보통 기대할 수 있는 변화:

- 노드당 key 수 증가
- 트리 높이 감소 가능
- split 빈도 감소 가능
- 탐색 시 방문 노드 수 감소 가능

하지만 동시에:

- 노드 하나가 커짐
- 메모리 낭비 가능
- 작은 데이터셋에서는 오히려 이득이 크지 않을 수 있음

즉 "무조건 큰 게 좋다"는 아니다.

---

## 12. key 타입을 바꾸면 뭐가 달라지나

### `int` -> `int64_t`

좋아지는 점:

- 더 큰 key 범위 지원
- overflow 위험 감소

나빠질 수 있는 점:

- key 크기 증가
- 노드당 key 수 감소 가능
- 트리 높이 증가 가능

즉:

```text
더 큰 숫자를 담고 싶다
-> key 타입을 키운다
-> 노드당 수용량이 줄 수도 있다
```

---

## 13. 왜 환경마다 값이 달라질 수 있나

이것도 중요하다.

`sizeof(long)`, `sizeof(off_t)`, `sizeof(pointer)`는
플랫폼마다 다를 수 있다.

즉 같은 소스라도

- macOS
- Linux
- 32-bit 환경
- 64-bit 환경

에서 실제 노드 수용량이 조금 달라질 수 있다.

그래서 진짜 구현을 볼 때는
항상

```c
sizeof(...)
```

기반 계산을 봐야 한다.

---

## 14. 초보자가 가장 많이 헷갈리는 포인트

### 1. "블록 크기 = 데이터 개수 제한"이라고 생각함

아니다.

블록 크기는
"노드 하나에 얼마나 들어가나"를 정한다.

데이터 전체 개수는
트리 높이가 올라가면서 더 많이 담을 수 있다.

### 2. "max key = 넣을 수 있는 데이터 개수"라고 생각함

아니다.

max key는
"key 숫자 범위 한계"에 가깝다.

예를 들어 key가 `id`라면,
id가 너무 커지지 않도록 해주는 타입 한계다.

### 3. "order는 무조건 하나의 뜻"이라고 생각함

아니다.

책, 블로그, 코드마다 다르게 쓸 수 있다.

항상 그 코드 안 정의를 먼저 봐야 한다.

### 4. "성능은 무조건 fanout 큰 게 최고"라고 생각함

그것도 아니다.

fanout이 커지면 높이는 낮아질 수 있지만,
노드 하나가 무거워지고
노드 내부 탐색 비용이나 캐시 효율이 달라질 수 있다.

## 15. 이 프로젝트 기준으로 보면 왜 중요한가

우리가 이 프로젝트에서 보는 구현은 크게 두 방향이 있다.

### 메모리 기반 구현

- 메모리에서 노드 관리
- 시작 시 재구축
- 디스크 페이지 관리 부담이 적음

여기서는

- key 타입
- 노드 구조
- split 규칙

같은 것이 더 중요하게 보인다.

### 디스크 기반 구현

- 파일 offset 사용
- block/page 크기 중요
- 부트 파일, free block, flush가 중요

여기서는

- block size
- offset 타입
- 저장 형식
- 파일 복원 방식

이 더 크게 중요해진다.

즉 같은 B-tree라도
메모리 기반과 디스크 기반은
신경 써야 하는 파라미터가 조금 다르다.
