# reum-011 디스크형 엔진 메모와 초심자용 해설

이 문서는 `reum-010` HTML을 읽다가 생길 수 있는 대표 질문을
프로그래밍 비전공자도 이해할 수 있게 풀어쓴 보조 문서다.

---

## 1. 한눈 요약

- 현재 비교표는 **다섯 개의 in-memory 트리 라이브러리**를 같은 규칙으로 비교한 결과다.
- 예전에 프로젝트에서 쓰던 디스크형 `thirdparty/bplustree`도 후보에 넣어보려 했지만,
  이번 재검증에서는 "assert로 바로 죽는 문제"보다
  **큰 value를 넣었을 때 값이 틀어지는 문제**가 먼저 확인됐다.
- 그래서 최종 표에는
  **끝까지 안정적으로 돌고 결과를 검증할 수 있었던 다섯 개 후보만** 남겼다.
- 즉 지금 보고 있는 순위표는
  "DB 전체 비교"가 아니라
  "트리 엔진 핵심 동작 비교"에 가깝다.

---

## 2. 처음 적어둔 `assert failure` 문구는 왜 다시 보게 되었나

초기 작업 메모에는

> `thirdparty/bplustree`가 100,000건 이상에서 assertion failure로 중단되었다

라고 적혀 있었다.

그런데 이 문장을 다시 검증해보니,
현재 환경에서는 그 assert를 같은 방식으로 재현하지 못했다.

대신 다시 붙여서 돌려본 결과는 이랬다.

### 다시 확인했을 때 관찰한 사실

1. raw `thirdparty/bplustree`를 직접 붙인 테스트에서
   `100,000`건 전체 시나리오는 통과했다.
2. `1,000,000`건에서도 dense 시나리오들은 통과했다.
3. 하지만 `sparse_rand_build_rand_get` 시나리오에서,
   큰 value를 넣었을 때 lookup 결과가 틀어졌다.

대표적으로 이런 식의 오류가 나왔다.

```text
expected 2564785620, got -1730181676
```

즉 "프로그램이 바로 죽는 assert"보다는
"값을 제대로 보존하지 못하는 현상"이 더 분명하게 잡혔다.

---

## 3. 그럼 이 라이브러리는 데이터 개수가 많으면 무조건 못 쓰는 건가

현재까지는 **그렇게 단정하기 어렵다**.

왜냐하면 우리가 다시 확인했을 때 문제는
"데이터 개수 자체"보다
"그 실험에서 넣은 value의 크기와 형태"와 더 관련 있어 보였기 때문이다.

### 왜 그렇게 보나

- 같은 `1,000,000`건이어도
  dense 시나리오는 지나갔다.
- 문제는 sparse 시나리오에서 먼저 보였다.
- 같은 sparse `1,000,000`건이라도
  value를 더 작은 범위로 두면 통과했다.

즉 지금 단계에서 더 정확한 설명은:

> "몇 개 이상은 절대 못 넣는다"라기보다  
> "이 벤치마크에서 사용한 큰 value 패턴을 안정적으로 보존하지 못했다"

에 가깝다.

---

## 4. 그래도 소스 코드 기준의 이론적 한계는 있나

있다. 다만 그건 "레코드 개수 상한"이라기보다
**key 타입 범위** 쪽 한계다.

`thirdparty/bplustree.h`를 보면 key 타입은 이렇게 선언돼 있다.

```c
typedef int key_t;
```

즉 key는 32-bit signed `int` 범위 안에 있어야 한다.

현재 sparse 시나리오는 대략 이런 공식을 쓴다.

```text
key = index * 257 + 11
```

그래서 이론적으로는
`INT_MAX`를 넘기지 않는 선까지만 안전하다.

현재 실험 러너 기준으로 계산하면
sparse 시나리오의 안전 상한은 대략 다음 정도다.

```text
8,355,968 records
```

이 숫자는
"그 이상은 반드시 실패한다"는 뜻이 아니라,
"key가 int 범위를 넘지 않게 유지하려면 이 근처가 상한"이라는 뜻이다.

---

## 5. 왜 큰 value에서 문제가 생겼다고 보나

이번 벤치마크는 lookup이 진짜로 맞는지도 검증하기 위해
key마다 value를 같이 저장한다.

예를 들면 이런 식이다.

```text
value = key * 13 + 7
```

문제는 sparse `1,000,000`건쯤 가면 key가 꽤 커져서,
이 value도 금방 커진다는 점이다.

예를 들어 key가 `197,291,201`이면:

```text
value = 197,291,201 * 13 + 7
      = 2,564,785,620
```

이 값은 32-bit signed 정수 최대값(`2,147,483,647`)보다 크다.

이번 재현에서는 바로 이 근처에서
lookup 결과가 음수로 뒤틀리는 현상을 봤다.

즉 실험상으로는
"큰 value를 안정적으로 왕복 저장/조회하지 못한다"는 식으로 보였다.

## 6. 파일 기반 인덱스가 쓰던 디스크형 엔진에 다른 다섯 라이브러리를 넣을 수 있나

### 짧은 답

가능은 하다.
하지만 **지금 벤치마크와 똑같지는 않다**.

### 왜 다르나

지금 비교표는
라이브러리의 핵심 동작을 같은 인터페이스로 맞춘 것이다.

즉 지금은 대체로 이런 함수만 비교했다.

- create
- insert
- get
- destroy

그런데 파일 기반 인덱스 엔진 자리에 다른 라이브러리를 넣는다는 건
트리 자체만이 아니라 아래 문제까지 같이 풀어야 한다.

- 파일을 어떻게 열지
- 종료 시 어떻게 저장할지
- 다시 시작했을 때 어떻게 복원할지
- 노드를 페이지처럼 디스크에 어떻게 배치할지
- free block, boot file, metadata를 어떻게 관리할지

즉 이건
**트리 라이브러리 교체**를 넘어서
**지속성 계층(persistence layer) 설계**까지 들어가는 일이다.

---

## 7. 그럼 어떤 방식으로 붙일 수 있나

보통 세 가지 큰 방식이 있다.

### 방식 A. 시작할 때 메모리로 올리고, 끝날 때 저장하기

가장 쉬운 방식이다.

흐름은 이렇다.

1. 프로그램 시작
2. 파일에서 key-value 목록 읽기
3. 메모리 트리 만들기
4. 작업 중에는 메모리 트리 사용
5. 종료 시 파일로 다시 저장

장점:

- 구현이 가장 쉽다
- 기존 in-memory 라이브러리를 재사용하기 좋다

단점:

- 진짜 디스크 기반 트리와는 다르다
- 시작/종료 비용이 크다

### 방식 B. 데이터 파일을 기준으로 매번 인덱스를 재구축하기

현재 `memory_bptree` 쪽 설계가 이쪽에 가깝다.

1. CSV나 row 파일이 진짜 데이터
2. 시작할 때 전체 스캔
3. 인덱스는 메모리에서 다시 만듦

장점:

- 구조가 단순하다
- 구현과 설명이 쉽다

단점:

- 시작 시간이 길어진다
- 인덱스를 영속 저장하는 실험은 아니다

### 방식 C. 노드 자체를 디스크 페이지처럼 저장하기

이게 "진짜 디스크형 인덱스"에 더 가깝다.

1. 노드를 메모리 구조체로만 두지 않음
2. 페이지 번호나 파일 offset로 연결
3. 필요할 때만 읽고 수정 후 다시 씀

장점:

- 실제 DBMS 구조와 더 비슷하다
- 대용량에서 의미 있는 실험이 가능하다

단점:

- 구현 난도가 크게 올라간다
- 라이브러리가 메모리 전용이면 wrapper가 아니라 거의 재설계 수준이 된다

## 8. 다섯 라이브러리별로 보면 어떤가

### `reum memory_bptree`

- 현재 구현은 메모리 전용이다.
- 파일 기반으로 쓰려면 별도 저장/복원 계층이 필요하다.

### `habedi/bptree`

- 메모리 기반 B+ tree로 쓰기 좋다.
- 디스크형으로 쓰려면 wrapper 또는 serializer를 따로 만들어야 한다.

### `tidwall/btree.c`

- 단일 C 파일이라 실험용으로 붙이기 쉽다.
- 하지만 이것도 기본은 메모리 트리다.

### `Kronuz/cpp-btree`

- 성능은 강했지만 역시 기본은 in-memory 컨테이너다.
- 디스크 페이지 관리까지 해주지는 않는다.

### `frozenca/BTree`

- 이번 비교에서는 in-memory 쪽만 썼다.
- 하지만 라이브러리 자체는 disk/mmap 관련 기능도 문서에 있어서,
  파일 기반 실험으로 확장할 때는 다른 후보보다 더 유리할 수 있다.

즉
"지금 다섯 개를 그대로 파일 기반 엔진 자리에 꽂는다"는 느낌보다는,
"각 라이브러리 위에 별도 디스크 계층을 설계한다"는 쪽이 더 정확하다.

---

## 9. 그럼 똑같은 벤치마크를 할 수 있나

완전히 똑같지는 않다.

왜냐하면 지금 벤치마크는
트리 핵심 처리량을 보려는 실험이고,
파일 기반 벤치마크는
디스크 I/O와 복원 비용까지 섞이기 때문이다.

그래서 더 공정하게 하려면
실험을 두 종류로 나누는 것이 좋다.

### 실험 A. 코어 트리 비교

- 지금 `reum-010`이 하는 일
- 메모리 위에서 insert/get만 공정하게 비교

### 실험 B. 파일 기반 인덱스 통합 비교

- 파일 열기
- 복원
- 인덱스 유지
- 종료 후 재시작
- 필요하면 range query

까지 포함해서 비교

이렇게 나누면
"트리 자체가 빠른가?"
와
"DB 인덱스로 붙였을 때 전체 구조가 좋은가?"
를 분리해서 볼 수 있다.

## 10. 발표나 문서에서 이렇게 말하면 덜 헷갈린다

아래 문장이 현재 상황을 가장 정확하게 설명한다.

> 예전 파일 기반 인덱스에서 쓰던 `thirdparty/bplustree`도 비교 후보로 검토했지만,  
> 이번 재검증에서는 대용량 sparse 시나리오에서 큰 value를 안정적으로 보존하지 못하는 문제가 먼저 관찰되었다.  
> 따라서 최종 비교표는 끝까지 안정적으로 검증 가능한 in-memory 구현 다섯 개만 포함한다.
